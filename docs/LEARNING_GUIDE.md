## Section 0: Environment Setup

### What it does
Three Docker services collaborate:
- **`db`** — PostgreSQL 16 with the `pgvector` extension (UUID + JSONB support)
- **`api`** — FastAPI; on startup, runs `alembic upgrade head` automatically before accepting traffic
- **`frontend`** — Next.js 14; proxies `/api/*` to the FastAPI container via a rewrite rule

### Where to read
| File | What to look for |
|------|-----------------|
| `docker-compose.yml` | Service dependencies (`depends_on`), port mappings, env vars |
| `apps/api/app/main.py` | `lifespan()` function — Alembic is called here before the app starts |
| `apps/frontend/next.config.ts` | `rewrites()` — explains why frontend calls `/api/...` and it reaches FastAPI |

### Manual experiment
```bash
cp .env.example .env          # add your ANTHROPIC_API_KEY
docker compose up --build     # watch the logs — Alembic migration runs on first boot

# Health check
curl http://localhost:8000/health
# → {"status": "ok"}

# Confirm the 5 tables were created
docker compose exec db psql -U fineas -d fineas -c "\dt"
# → assets, conversations, goals, projections, snapshots

# Seed real data from the Excel file
docker compose exec api python scripts/migrate_excel.py
# → visit http://localhost:3001 — net worth should be populated
```

### Key insight
The `INTERNAL_API_URL=http://api:8000` environment variable lets server-side Next.js
components call FastAPI via Docker DNS. Browser-side calls use the public
`NEXT_PUBLIC_API_URL=http://localhost:8000` instead. This is why `lib/api.ts` has two
different URL resolution paths.

## Section 1: Data Model

### What it does
Five SQLAlchemy 2.0 models form the core schema. Everything else reads/writes through them.

```
Asset  ──< Snapshot     (one asset, many monthly snapshots)
Goal   ──< Projection   (one goal, many projection runs)
Conversation            (standalone; stores message history as JSONB array)
```

### Where to read
| File | Key lines | What to notice |
|------|-----------|----------------|
| `apps/api/app/models/asset.py` | 14–34 | `server_default=text("gen_random_uuid()")` — UUID generated by Postgres, not Python |
| `apps/api/app/models/snapshot.py` | whole file | Unique constraint `(asset_id, date)` — enforced at DB level |
| `apps/api/app/models/goal.py` | whole file | `asset_scope` is `JSONB` — flexible filter (e.g. `{"exclude": ["p2p_lending"]}`) |
| `apps/api/app/models/projection.py` | whole file | `params` and `results` are both `JSONB` — schema-less storage for projection data |
| `apps/api/app/models/conversation.py` | whole file | `messages` is `ARRAY(JSONB)` — the full chat history in one column |
| `apps/api/alembic/versions/594ebbe359eb_initial_schema.py` | whole file | The actual DDL; compare to the models to see the 1:1 mapping |

### Manual experiment
```bash
# Connect to the DB directly and inspect the schema
docker compose exec db psql -U fineas -d fineas

-- See table columns and types
\d assets
\d snapshots

-- After seeding: query real data
SELECT name, asset_type, platform FROM assets ORDER BY name;
SELECT a.name, s.date, s.amount FROM snapshots s JOIN assets a ON a.id = s.asset_id ORDER BY s.date DESC LIMIT 10;

-- The unique constraint in action
INSERT INTO snapshots (asset_id, date, amount) VALUES ('<any-asset-uuid>', '2026-01-01', 999);
-- run it twice → the second one will fail with a unique violation
```

### Key patterns
- **UUIDs from Postgres** (`gen_random_uuid()`) — Python never generates PKs; the DB does
- **Soft deletes via `is_active`** — used on `Asset` and `Goal`; rows are never physically deleted
- **`JSONB` for flexibility** — `metadata_` on Asset, `asset_scope` on Goal, `params/results` on Projection
- **`cascade="all, delete-orphan"`** on `Asset.snapshots` — deleting an asset deletes its snapshots

## Section 2: Projection Service (Pure Business Logic)

### What it does
`services/projection.py` is the only piece of code with **no external dependencies** —
no HTTP, no database, just math. It computes a deterministic FIRE trajectory using
this formula:

```
FV(t) = PV × (1 + r)^t  +  C × [((1 + r)^t - 1) / r]
```

Where `PV` = current portfolio value, `r` = monthly real return rate, `C` = monthly
contribution, `t` = number of months.

### Where to read
| File | Lines | What to notice |
|------|-------|----------------|
| `apps/api/app/services/projection.py` | 25–32 | `REAL_RETURNS` dict — the assumed real (inflation-adjusted) rates per asset type |
| `apps/api/app/services/projection.py` | 62–71 | `_compute_weighted_return()` — pure function, no DB. Weighted average of per-type returns |
| `apps/api/app/services/projection.py` | 91–117 | The main loop — one iteration per year, applying the formula above |
| `apps/api/app/services/projection.py` | 112–117 | `target_hit_year` detection — first year FV crosses the goal amount |
| `apps/api/app/schemas/projection.py` | whole file | `CompoundParams`, `CompoundResult`, `CompoundYearlyPoint` — the input/output shapes |

### Manual experiment
```bash

>>> # 100% stocks portfolio
>>> _compute_weighted_return({"stocks": 50_000}, 50_000)

>>> # Mixed portfolio: 60% stocks, 40% bonds
>>> _compute_weighted_return({"stocks": 60_000, "bonds": 40_000}, 100_000)

>>> # Simulate the formula manually
>>> pv, c, r_annual, years = 50_000, 750, 0.05, 20
>>> r = r_annual / 12
>>> t = years * 12

### Key patterns
- **`_compute_weighted_return`** is a pure function — takes a `dict[str, float]` and a total,
  returns a float. Test this in total isolation from the DB.
- **Edge case**: when `monthly_r == 0` (cash-only portfolio), the formula simplifies to
  `FV = PV + C * t` (line 95–96) — avoids division by zero.
- **`_get_current_portfolio`** (lines 37–58) is the only DB-touching part. It's a private
  helper; the testable math is all in `_compute_weighted_return` and the loop.

## Section 3: Portfolio Tools

### What it does
`tools/portfolio.py` bridges the agent layer with the database. It's not a class — it's a
**factory function** (`make_portfolio_tools`) that returns a `dict` of async closures, each
with a database session injected via closure.

Four tools:
| Tool | What it does |
|------|-------------|
| `get_current_holdings()` | Latest snapshot per active asset; returns dict keyed by asset name |
| `get_asset_by_name(query)` | Fuzzy-matches a user's text to known asset names (exact → substring → Levenshtein) |
| `get_net_worth_summary()` | Total + ranked holdings list |
| `batch_update_snapshots(updates)` | Upsert snapshots for a list of `{asset_id, date, amount}` |

### Where to read
| File | Lines | What to notice |
|------|-------|----------------|
| `apps/api/app/tools/portfolio.py` | 52–85 | `AssetDelta` — anomaly logic lives here; >25% change or negative value sets `is_anomaly=True` |
| `apps/api/app/tools/portfolio.py` | 101–108 | `make_portfolio_tools` factory pattern — session injected once, all tools share it |
| `apps/api/app/tools/portfolio.py` | 147–180 | `get_asset_by_name` — three-pass matching: exact → substring → Levenshtein; returns `confidence` score |
| `apps/api/app/tools/portfolio.py` | 191–238 | `batch_update_snapshots` — error-isolated per update; one failure doesn't abort the rest |

### Manual experiment

...         print(match)  # → {"asset_name": "Xtrackers EUR Overnight", "confidence": ...}
...         match2 = await tools["get_asset_by_name"]("Scalabl stocks")
...         print(match2)
...         # Net worth
...         summary = await tools["get_net_worth_summary"]()
...         print(f"Total: €{summary['total_net_worth']:,.0f}")

>>> asyncio.run(try_tools())
```

Also try the anomaly logic directly:
```bash
>>> from app.tools.portfolio import AssetDelta
>>> d = AssetDelta("Stocks", "some-uuid", old_amount=10_000, new_amount=13_000)
>>> d.is_anomaly, d.delta_pct   # → True, 30.0  (>25%)
>>> d2 = AssetDelta("Stocks", "some-uuid", old_amount=10_000, new_amount=10_500)
>>> d2.is_anomaly                # → False
```

### Key patterns
- **Closure factory** — `make_portfolio_tools(session)` creates closures that capture the
  session. This is the pattern used by all LangGraph nodes that need DB access.
- **Levenshtein fallback** — only kicks in when exact and substring matching fail;
  `confidence < 0.5` → returns an error dict, not a match. This is the user-facing
  "I don't know what asset you mean" path.
- **Per-update error isolation** in `batch_update_snapshots` — the `try/except` around each
  update + `await session.rollback()` means one bad update doesn't corrupt the rest.

## Section 4: API Routes

### What it does
Five FastAPI routers expose the database and business logic over HTTP and WebSocket.
All are registered in `main.py` under the `/api` prefix.

| Router | Prefix | Key endpoints |
|--------|--------|---------------|
| `assets` | `/api/assets` | CRUD; `?include_inactive=true` to show inactive |

| `projections` | `/api/projections` | `POST /run` (runs projection service) |

### Manual experiment — fill-forward SQL
The history endpoint has non-obvious logic. Understand it by reading the SQL (lines 84–113 of `snapshots.py`):

```sql
-- The CTE only includes dates where >1 asset has a snapshot
-- (avoids charting a day where only one asset was recorded)
HAVING COUNT(DISTINCT s.asset_id) > 1

-- Then cross-joins every date with every asset
-- The correlated subquery finds the most recent snapshot on or before that date
-- This is fill-forward: if an asset has no snapshot on a date, use the last known value
SELECT s2.amount FROM snapshots s2
WHERE s2.asset_id = a.id AND s2.date <= sd.date
ORDER BY s2.date DESC LIMIT 1
```

Verify this by adding a snapshot for just one asset on an isolated date — it should NOT appear
in the history result.

## Section 5: Agents

### Overview
Two agent files coordinate the chat experience:

```
WebSocket message received
       │
       ▼
orchestrator.handle_message()
       │
       ├─ intent="update" ──► update_agent graph (LangGraph)
       │                              │
       │                              ├─ parse_input (LLM call)
       │                              ├─ fetch_current (DB)
       │                              ├─ compute_deltas (fuzzy match)
       │                              ├─ present_confirmation (format text)
       │                              └─ [INTERRUPT] ← graph pauses here
       │                                      │
       │                         WebSocket sends confirmation_request
       │                                      │
       │                         User clicks Confirm / Cancel
       │                                      │
       │                         handle_confirmation() resumes graph
       │                              ├─ write_updates (DB write)
       │                              └─ respond (new net worth)
       │
       └─ other intent ──► _general_response() (single LLM call, no graph)
```

### Section 5a: Orchestrator

#### Where to read

### Section 5b: Update Agent (LangGraph)

#### Where to read
| File | Lines | What to notice |
|------|-------|----------------|
| `apps/api/app/agents/update_agent.py` | 30–31 | `_CHECKPOINTER = MemorySaver()` — module-level singleton; this is how state survives the `handle_confirmation()` resume |
| `apps/api/app/agents/update_agent.py` | 44–53 | `UpdateAgentState` TypedDict — the full state that flows between nodes |
| `apps/api/app/agents/update_agent.py` | 60–87 | `SYSTEM_PROMPT` — the exact instruction set given to Claude for parsing |
| `apps/api/app/agents/update_agent.py` | 102–127 | `parse_input` node — LLM call → JSON parse → list of `ParsedUpdate` |
| `apps/api/app/agents/update_agent.py` | 138–198 | `compute_deltas` — fuzzy-matches parsed updates to real assets, builds `AssetDelta` list |
| `apps/api/app/agents/update_agent.py` | 227–248 | `write_updates` — first checks `user_confirmed`; skips DB write if False |
| `apps/api/app/agents/update_agent.py` | 292–333 | `build_update_graph()` — graph construction; notice `interrupt_before=["write_updates"]` |
| `apps/api/app/agents/update_agent.py` | 276–285 | `route_after_confirmation` — if no deltas or error, goes to `END`, not `write_updates` |

#### The interrupt mechanism — read carefully
```
graph.astream(initial_state, config)  ← first call (handle_message)
       │  runs: parse_input → fetch_current → compute_deltas → present_confirmation
       │  STOPS at write_updates (interrupt_before)
       │  checkpoint is saved to _CHECKPOINTER under thread_id = conversation_id
       ▼
  graph pauses, stream ends

graph.aupdate_state(config, {"user_confirmed": True})  ← injects user decision
graph.astream(None, config)  ← second call (handle_confirmation)
       │  resumes from checkpoint: write_updates → respond → END
       ▼
  done
```

The key insight: the graph is **rebuilt** with a new session in `handle_confirmation()`, but
`_CHECKPOINTER` is module-level, so the checkpoint state from the first run is still there.
That's why the second `build_update_graph(new_session)` call can resume from where the first left off.

## Section 6: Frontend

### What it does
Four pages, each with a distinct purpose:

| Route | Component | Data source |
|-------|-----------|-------------|
| `/` | Dashboard | Server component; SSR fetch of `/api/snapshots/latest` and `/api/snapshots/history` |
| `/chat` | Chat interface | Client component; WebSocket via `useChat` hook |
| `/projections` | Projection view | Server component; calls `POST /api/projections/run` |
| `/data` | Data management | Server + client; asset list SSR, snapshot CRUD client-side |

### Where to read
| File | What to notice |
|------|---------------|
| `apps/frontend/lib/api.ts` | `apiFetch()` helper; note the `INTERNAL_API_URL` vs relative URL split for server vs client |
| `apps/frontend/lib/hooks/useChat.ts` | Full WebSocket lifecycle; `sendMessage()`, `confirm()`, message state |
| `apps/frontend/lib/websocket.ts` | URL resolution for localhost vs Docker vs GitHub Codespaces |
| `apps/frontend/components/chat/ConfirmationCard.tsx` | Inline editing of amounts before confirm; maps `delta.asset_name` to editable inputs |
| `apps/frontend/components/dashboard/NetWorthChart.tsx` | Recharts `AreaChart`; EUR locale formatting |
| `apps/frontend/app/page.tsx` | Server component pattern — async function that calls API at render time |
| `apps/frontend/next.config.ts` | `rewrites()` — `/api/:path*` → `http://api:8000/api/:path*` |

### Manual walkthrough

**Dashboard**
1. Open `http://localhost:3001`
2. Confirm net worth hero shows the correct total
3. Hover over the area chart — tooltips should show EUR-formatted values
4. The donut chart should have slices for each asset type in your portfolio
5. The holdings table should show allocation % summing to ~100%

**Chat**
1. Navigate to `/chat`
2. Look for the green dot — it indicates WebSocket is connected
3. Type: `"My Scalable Capital stocks are now 47000"`
4. Watch: "Thinking…" message appears (agent is running parse_input + fetch_current + compute_deltas)
5. A confirmation card appears showing before → after values
6. If the change is >25%, an amber "anomaly" badge appears
7. Try editing the amount in the confirmation card before confirming
8. Click Confirm → watch for "update_complete" message and new net worth

**Data Management**
1. Navigate to `/data`
2. Left sidebar: click the toggle on an asset to deactivate it → it should disappear from dashboard
3. Right panel: select an asset, add a snapshot for today's date

### Key patterns to understand
- **`useChat` hook** (`lib/hooks/useChat.ts`) — study how it maps incoming WebSocket message
  types to UI state: `"thinking"` → shows bouncing dots, `"confirmation_request"` → renders
  `ConfirmationCard`, `"update_complete"` → dismisses card
- **Server vs Client Components** — `app/page.tsx` is `async` and calls `getPortfolioSummary()`
  directly (no client-side fetch). `ChatPanel.tsx` has `"use client"` and uses the hook.
  This is the Next.js App Router pattern.
